## Коротко

Статья разбирает, как Microsoft построила Microsoft Learn MCP Server: удаленный MCP-сервер, который дает AI-агентам доступ к официальной и актуальной документации Microsoft Learn. Для проекта это хороший практический кейс инженерного мастерства: команда не просто открыла API, а спроектировала агентоориентированный контракт, обернула существующую retrieval-инфраструктуру в стандартный протокол и столкнулась с реальными проблемами эксплуатации публичного сервиса.

## Основной тезис

Главная мысль статьи: инструмент для AI-агентов нельзя проектировать как тонкую копию внутреннего API. Его нужно проектировать как стабильный, понятный и эволюционирующий контракт, который агент может обнаружить, понять и использовать в собственном workflow.

## Полезные идеи

- MCP-сервер выступает как слой между AI-агентами и внутренней knowledge/retrieval-инфраструктурой. Это архитектурная граница: снаружи агент видит небольшое число понятных инструментов, внутри остаются retrieval-параметры, индексы, ранжирование и инфраструктурная сложность.
- Команда не стала напрямую отдавать агентам все параметры внутреннего сервиса. Вместо этого она сжала контракт до операций, похожих на человеческий паттерн "найти и прочитать": поиск документации, получение полной страницы и поиск примеров кода.
- Удаленный MCP-сервер оказался не просто JSON-RPC-оберткой, а обычной распределенной системой: появились вопросы масштабирования, регионов, CORS, session affinity, statelessness и защиты данных.
- Описания инструментов и параметров стали частью пользовательского опыта агента. Небольшие изменения формулировок заметно меняли поведение клиентов, поэтому описания пришлось улучшать через автоматизированную оценку и метрики успешности.
- Пара search + fetch работает лучше как композиция, чем как один слишком широкий инструмент: поиск находит кандидатов, fetch дает полный Markdown-контекст для grounding и цитирования.
- Даже при динамическом discovery часть клиентов все равно хардкодит схемы. Это делает эволюцию публичного контракта хрупкой и требует обратной совместимости, deprecation window и поддержки старых имен параметров.
- Данные использования показали реальные сценарии: coding tasks, explanation, troubleshooting. Команда использовала эти паттерны, чтобы приоритизировать изменения в описаниях инструментов и retrieval-поведении.

## Инженерный разбор

- Контекст и задача: AI-агентам, включая GitHub Copilot и другие MCP-клиенты, нужен надежный доступ к официальной документации Microsoft Learn без ручного поиска, scraping и отдельных интеграций для каждого клиента.
- Ограничения: публичный удаленный сервис, разные MCP-клиенты, необходимость актуального knowledge grounding, существующая Learn knowledge service, совместимость с протоколом MCP, эксплуатация в облаке и эволюция контракта без массовых поломок.
- Инженерные решения: использовать MCP как стандартный агентоориентированный протокол; разместить remote MCP server перед Learn knowledge service; реализовать транспорт и session handling через официальный C# SDK; разместить сервис на Azure App Service; опереться на content vector store и существующую RAG-инфраструктуру Ask Learn.
- Инструментальный контракт: предоставить три инструмента - поиск документации, получение полной статьи и поиск примеров кода. Это уменьшает когнитивную и контрактную сложность для агента.
- Что получилось хорошо: внутренние retrieval-настройки не протекли наружу; контракт стал ближе к workflow агента; существующая knowledge-инфраструктура была переиспользована; динамическое discovery позволило клиентам узнавать schema и tools на старте сессии; композиция search/fetch улучшила groundedness и качество цитирования.
- Что получилось плохо или рискованно: часть клиентов все равно обращалась с MCP tools как с жестко зафиксированным API; переименование параметра сломало заметную долю запросов; поведение агента оказалось чувствительным к формулировкам описаний; публичный remote server принес эксплуатационную сложность распределенной системы.
- Трудности и неожиданные проблемы: cross-region deployment, dynamic scaling, CORS, session affinity, statelessness, data protection, backward compatibility, hardcoded callers, tuning tool descriptions на основании поведения реальных агентов.
- Архитектурные уроки: внешний контракт для агента должен быть меньше и стабильнее внутреннего API; описание инструмента - это часть интерфейса; публичная схема должна эволюционировать защитно; в agent-facing systems важна не только retrieval-логика, но и управляемость протокола, наблюдаемость поведения клиентов и процесс безопасного изменения.
- Переносимые приемы: оборачивать сложный внутренний API в task-oriented tools; не выносить все параметры наружу; проектировать tools по workflow пользователя или агента; строить evaluation loop для описаний инструментов; поддерживать старые параметры во время deprecation; собирать агрегированные usage patterns и улучшать контракт по реальному поведению.

## Инсайты для поиска темы

- Для [[Инженерное мастерство|инженерного мастерства]] статья показывает мастерство как способность спроектировать правильную границу: не максимальная мощность API, а правильная форма интерфейса под потребителя.
- Для [[Архитектура ПО|архитектуры ПО]] важен тезис: архитектурный слой может быть не вычислительным, а протокольно-продуктовым. MCP-сервер превращает внутреннюю retrieval-систему в discoverable tool surface для агентов.
- Для [[Теория лимитов|теории лимитов]] источник дает пример лимита контракта: если наружу вынести слишком много параметров, система становится гибкой, но хрупкой; если слишком мало - агент не сможет решать задачи. Мастерство в том, чтобы выбрать рабочую границу.
- Для [[RAG]] статья дополняет production-перспективу: качество grounding зависит не только от индекса и retrieval, но и от того, как агент вызывает инструменты и какие инструкции получает через tool descriptions.
- Для [[Наблюдаемость|наблюдаемости]] интересен переход от обычных технических метрик к наблюдению поведения агентов: tool activation, success metrics, usage intents и влияние формулировок на downstream behavior.

## Проверка идей и контргипотезы

- Гипотеза: агентные инструменты нужно проектировать не как API-обертки, а как task-oriented контракты. Контргипотеза: для некоторых клиентов низкоуровневый API может быть полезнее, если они способны оптимизировать retrieval самостоятельно.
- Гипотеза: tool descriptions являются частью интерфейса системы. Контргипотеза: их влияние может быть сильно зависимо от конкретных LLM, клиентов и задач, поэтому нужны сравнительные эксперименты.
- Гипотеза: MCP снижает стоимость интеграции и ломкость клиентов. Контргипотеза: на практике клиенты все равно могут хардкодить схемы, поэтому сам протокол не отменяет дисциплину backward compatibility.
- Проверить дальше: исследования по tool use, tool calling evaluation, protocol evolution, API usability, RAG evaluation и наблюдаемости agent-facing systems.
