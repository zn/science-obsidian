## Коротко

Статья Uber Engineering описывает, как компания перестраивала identity and access stack для внутренних AI-агентов: агентам выдали проверяемую идентичность, цепочку делегирования начали переносить через все hops, а MCP Gateway сделали точкой применения политик. Для проекта это сильный инженерный кейс о том, как рост автономности меняет границы архитектуры, наблюдаемости и ответственности.

## Основной тезис

Главная мысль источника: в агентных системах недостаточно знать, какой сервис вызвал API. Нужно сохранять проверяемую цепочку "кто действовал, через какого агента, с каким делегированием и для какого downstream-действия", иначе система теряет аудит, fine-grained authorization и управляемость инцидентов.

## Полезные идеи

- Традиционная IAM-модель "человек плюс workload/service account" плохо описывает agency. Агент действует не просто как workload, а как сущность, которой делегирована задача от человека или другого агента.
- Главная потеря в multi-agent workflow - не только потеря credentials, а потеря provenance: downstream-система видит generic service identity и не видит исходного пользователя, промежуточных агентов и контекст решения.
- Uber разделяет workload identity и agent identity. Workload подтверждается SPIRE/SVID, а Security Token Service проверяет, что конкретный agent_id разрешен на этом workload, и выдает короткоживущий JWT для следующего hop.
- Токен выдается per-hop и audience-scoped: токен для вызова одного агента или gateway не должен быть переиспользован для другой системы. Это делает границу действия технически выраженной.
- Actor chain в токене переносит историю делегирования: пользователь, первый агент, следующий агент, tool gateway. За счет этого authorization может учитывать не только последнего caller, но и всю проверенную линию действия.
- MCP Gateway выступает policy enforcement point для tool invocations: он проверяет токен, применяет tool-level политики, redaction и риск-классификацию.
- Secure-by-default путь сделали частью developer experience: standardized A2A client автоматизирует token exchange и propagation actor chain, чтобы инженерам не приходилось вручную реализовывать правильный путь.
- Наблюдаемость для агентных систем должна показывать не только технический trace, но и акторов, решения авторизации, tool calls и security context.

## Инженерный разбор

- Контекст и задача: в Uber появились production-grade AI-агенты, которые вызывают другие агенты, MCP tools и внутренние системы. Нужно сохранить accountability: кто инициировал действие, какой агент действовал, почему действие было разрешено.
- Ограничения: тысячи microservices, существующий Zero Trust stack, Kubernetes workloads, внутренний Agent platform, MCP поверх существующих service APIs, потребность в низкой задержке, миграция уже существующих агентов и необходимость не замедлить developer velocity.
- Инженерные решения: Agent Registry как source of truth для agent-to-workload binding; STS как trust broker; короткоживущие scoped JWT на каждый hop; actor chain внутри token claims; MCP Gateway как централизованная точка enforcement; AI Gateway и AI Guard для model-side рисков; standardized A2A client как paved path.
- Что получилось хорошо: identity стала проверяемой и переносимой; downstream-системы получили контекст всей цепочки; политики можно выражать с учетом пользователя и агента; audit trail стал полнее; secure path встроили в SDK/client, а не оставили ручной дисциплиной.
- Что получилось плохо или рискованно: решение сильно зависит от внутренней платформы, SDK и дисциплины регистрации агентов; actor chain увеличивает сложность токенов и политик; при ошибках в registry, claims или gateway enforcement blast radius может быть высоким; статья почти не показывает негативные кейсы миграции.
- Трудности и неожиданные проблемы: уже существующие агенты нужно переводить на новый A2A client; provenance должен переноситься на application layer, а не только внешним proxy; per-hop exchange добавляет latency overhead; policy engine должен понимать смешанный контекст "human + agent + intent/tool".
- Архитектурные уроки: автономность требует новой границы identity; атрибуция - это не логирование постфактум, а часть runtime-протокола; безопасность должна быть встроена в стандартный путь разработки; gateway полезен, когда он не только проксирует, но и выражает политики на уровне инструмента.
- Переносимые приемы: отделять workload identity от agent identity; делать токены audience-scoped и short-lived; переносить actor chain end-to-end; проектировать SDK/client так, чтобы правильное поведение было default; строить observability вокруг actor chain, authorization decision и tool invocation.

## Инсайты для поиска темы

- Для [[Теория лимитов|теории лимитов]] источник показывает новый тип лимита: граница допустимого действия определяется не только API-контрактом, но и проверяемой цепочкой делегирования. Если цепочка теряется, система становится функционально мощной, но управленчески слепой.
- Для [[Архитектура ПО|архитектуры ПО]] важен тезис: identity propagation - архитектурный механизм, а не вспомогательная security-фича. Он определяет, какие компоненты могут доверять действиям друг друга.
- Для [[Наблюдаемость|наблюдаемости]] кейс усиливает идею, что наблюдаемость расширяет управляемость: actor-chain trace делает видимым источник действия и позволяет разбирать инциденты без ручного склеивания логов.
- Для [[Инженерное мастерство|инженерного мастерства]] статья показывает мастерство платформенного решения: вместо призыва "разработчики должны правильно передавать контекст" команда строит paved path через SDK, STS и A2A client.
- Для [[MCP]] источник добавляет security-слой: MCP Gateway является не только протокольной границей tools, но и точкой авторизации, redaction и аудита агентных действий.

- [[Теория лимитов]]: поддерживает ведущую гипотезу о границах как архитектурной форме. Здесь граница выражена через identity, audience, token lifetime, actor chain и policy enforcement.
- [[Инженерное мастерство]]: сильный практический кейс о том, как инженеры превращают риск автономности в архитектурный протокол и developer experience.
- [[Кандидатские темы]]: усиливает темы про системный анализ архитектурных лимитов интеллектуальных программных систем и наблюдаемость как механизм расширения управляемости.
- [[Искусственный интеллект]]: полезен для ветки agentic AI systems, где ключевым объектом анализа становится не модель, а сеть действий, инструментов, полномочий и traceability.

## Проверка идей и контргипотезы

- Научно проверяемое утверждение: перенос actor chain через agent hops повышает полноту audit trail и точность policy enforcement в multi-agent workflows. Проверка: сравнить полноту атрибуции, число ambiguous actions, скорость incident response, false allow/deny и overhead до и после внедрения.
- Контргипотеза: для части систем достаточно централизованного gateway logging или session tracing без криптографического actor chain в каждом hop. Это может быть дешевле и проще, если workflow короткий, число tools мало, а downstream-системы не принимают самостоятельные policy decisions.
- Инженерная эвристика: secure path должен быть paved path - SDK/client должен автоматически делать token exchange и propagation context. Возможное ослабление: сильная зависимость от SDK может снижать portability, усложнять нестандартные агенты и создавать platform lock-in.
- Философская метафора: "agency" можно понимать как переносимую границу ответственности. Метафора полезна для мышления, но для научной темы ее нужно переводить в проверяемые понятия: субъект действия, делегирование, provenance, authorization decision, audit trail.
- Статус идеи: перспективная для инженерного и системно-аналитического трека, но требует источников по IAM, provenance, distributed tracing, access control, Zero Trust, OAuth token exchange и agent security standards.

## Вопросы

- Как формально описать отличие service identity, workload identity и agent identity?
- Можно ли рассматривать actor chain как архитектурный контракт между агентной системой и downstream-системами?
- Какие метрики доказывают, что identity propagation реально улучшает управляемость, а не только добавляет security-сложность?
- Где проходит лимит actor chain: размер токена, latency, privacy, сложность policy language, переносимость между организациями?
- Как учитывать intent агента: достаточно ли actor chain, или нужен отдельный claim о задаче, session, risk и human approval?
- Какие стандарты и научные работы читать дальше: SPIFFE/SPIRE, OAuth 2.0 Token Exchange, WIMSE, provenance models, access control for autonomous agents, distributed tracing?
